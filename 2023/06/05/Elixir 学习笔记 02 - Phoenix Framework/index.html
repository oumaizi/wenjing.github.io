Model-View-Controller

  • Model: Manages the data and business logic of the application
  • View: Handles layout and display, rendering the visual layout for the user
  • Controller: Handles requests and manipulates the model/view to respond to the user.

Plug

  • defines the way_you build web apps in Elixir and
  • it provides you with tools to write apps that are understood by web servers.

Directory structure

  • Directory structure — Phoenix v1.7.3
  • root.html.heex -> defines some meta information about our application, imports CSS styles, and renders our content using @inner_content
  • app.html.heex -> contains the header rendered on every page and renders flash messages used for displaying errors

Embedded Elixir

  • <%= elixir code%>
  • Business logic in the Phoenix framework refers to the core functionality of your application, organized into contexts as Elixir modules.
  • In Phoenix, we group common behavior into a context. Contexts expose an API for a set of related behavior. How we group, the behavior may change as the business logic of an application grows.

External and Internal Events

  • def handle_info() -> Internal Events: trigger inside the live function
  • def handle_event() -> External Events: trigger on the view layer

Bindings

  • phx-debounce="1000" -> emitting the event is delayed by the specified milliseconds1000
  • Bindings — Phoenix LiveView v0.19.0

Generator Command

  • mix phx.gen.context Boats Boat boats model:string type:string price:string image:string
  • The command takes the name of the context module (Boats) followed by the Ecto schema module (Boat) and its lowercase plural name used as the database table name (boats), and then any attributes and their types. We used a string type for the price attribute since prices are represented as a series of dollar signs ("$$$").

Memory Save using temporary_assgins

  • {:ok, socket, temporary_assgins: [keywork: []]} -> use it while simply fetching and rendering what’s effectively static data.
  • When you’re fetching data from a database (or any external data source), using temporary_assigns works great because you can just reload the data from the source again
  • It is happened after the initial render and between handling events.
